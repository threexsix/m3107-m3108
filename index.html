Организация занятий

  1. Всего будет 6 лабораторных работ. Гугл-диск
  2. Каждая лабораторная стоит по 10 баллов. По каждой лабораторной дедлайн на следующем занятии с момента выдачи. За сдачу после дедлайна можно получить максимум 50% от максимального числа баллов.
  3. Сдавать вперёд дедлайна можно.
  4. Баллы за лабораторные будут жить в этой табличке.
  5. Все вопросы можно задавать в чате телеграм-канала
  6. В начале каждого занятия мы будем просить вас дать нам табличку со списком на сдачу. Вы можете организовать очередь заранее и приходить на занятие в соответстии с ним. Например, если вы пятнадцатый в очереди, то вам имеет смысл прийти где-то ближе к началу второй пары. На занятиях не будет происходить ничего, кроме приёма лабораторной и ответов на возникшие вопросы, поэтому вы вполне можете прийти, сдать свою лабораторную и уйти.

Как проходит сдача

Сдача проходит посредством ревью Pull Request’a на GitHub
Для сдачи лабораторной работы необходимо сделать следующее:

  1. Создать свой аккаунт на https://github.com (у кого его ещё нет) и установить git у себя на компьютере (инструкция)
  2. Создать новый репозиторий.
  3. Добавить в список коллабораторов репозитория somov7 и BorisPetrianik.
    4.
  4. Сделайте Fork этого репозитория
  5. Открыть консоль/терминал, зайти в директорию, куда вы хотите клонировать репозиторий cd /path/to/directory, клонировать репозиторий себе git clone https://github.com/username/reponame.git, зайти в него cd reponame
  6. Создать новую ветку для вашей лабораторной: git checkout main; git checkout -b lab1
  7. Написать код, скомпилировать, запустить, убедиться, что всё работает
  8. Добавить файлы с исходным кодом в репозиторий: git add filename1.h filenam1.c filename2.c; git commit -m "commit message". Комментарий к коммиту желательно должен содержать краткое осмысленное описание того, что содержится в коммите. Пожалуйста, не добавляйте скомпилированные исполняемые или объектные файлы в репозиторий, только файлы с исходным кодом (.c и .h)
  9. Залейте свою ветку на гитхаб git push --set-upstream origin lab1
  10. Зайдите на GitHub и создайте новый Pull Request из вашего репозитория с вашей лабораторной в начальный репозиторий. Сделать это необходимо до среды 23:59 за три дня до пары, чтобы мы успели сделать ревью.
  11. В поле “Reviewers” добавьте преподавателей (somov7 и BorisPetrianik) в ревьюверы пулл-реквеста.
  12. Запишитесь на сдачу лабораторной с помощью google-формы. Все, кто не успеет к этому сроку, считаются пропустившими дедлайн. Даже если вы создали Pull Request, но не заполнили форму, не считается, что вы отправили лабораторную работу на проверку.
  13. Дождитесь ревью пулл-реквеста и придите на занятие, чтобы защитить лабораторную. На защите будут вопросы чтобы убедиться, что код написан действительно вами и вы понимаете что вы сами написали.
  14. После защиты у вас будет две недели для того, чтобы исправить недочёты. Если недочёты незначительные, вы всё ещё можете претендовать на полный балл за лабораторную после исправлений. Повторно защищать лабораторную после исправления недочётов не требуется, если мы в конкретном случае не скажем обратное. Максимум можно получить до +5 баллов от начальной оценки после ре-ревью, поэтому старайтесь писать сразу правильно. Делается максимум одно ре-ревью, бесконечно исправлять лабораторную нельзя, поэтому старайтесь сразу исправить все косяки и не наделать новых.
  15. После того как вы защитили лабораторную, баллы появятся в табличке. Чтобы эти баллы появились в ИСУ необходимо сделать отчёт. Отчёт нужно скинуть на google-диск, предварительно создав в нём папочку названную по ФИО. Названия файлов должны быть в виде Лабораторная работа №1, Иванов Иван Иванович, группа M310X, в формате .docx или .pdf. Можно заняться этим в конце семестра. Шаблон для титульного листа отчёта можно взять тут. Отчёт является скорее формальностью, тем не менее он должен быть содержательным, чтобы по нему можно было понять что вы сделали и какого результата добились: внутри должны быть пункты задания, объяснение, что вы делали и исходный код.

Если кому-то удобнее, то можете пользоваться встроенными средствами для системы контроля версий в вашей IDE или графическими клиентами по типу GitHub Desktop или GitKraken. Главное, чтоб вы создали репозиторий, создали Pull Request и призвали в него преподавателей на ревью.

FAQ

  1. А где вообще писать, какую IDE и компилятор использоват использовать?
    Если у вас уже есть своя любимая настроенная IDE, можете использовать её.
    1.1 Сначала установите компилятор.
    Если у вас какой-либо дистрибутив линукса, наверняка у вас уже есть компилятор языка Си из коробки. Можете это проверить, вбив в терминал команду gcc --version.
    Если у вас MacOS: brew install gcc
    Если у вас Windows: вот инструкция
    1.2 Установите себе IDE. Рекомендую использовать либо VSCode, либо CLion (лицензионный ключ должен быть где-то в ИСУ). Рекомендую всё таки использовать первое, так как, во-первых, вы уже используете VSCode на цифровой культуре, а во-вторых, CLion для данных лабораторных это overkill, или как говорят по-русски, “из пушки по воробьям”, она достаточно требовательная к железу и там много фичей которые вам не очень нужны. Если вы установили VSCode, то рекомендуем производить компиляцию ваших программ через терминал (поисковый запрос “how to compile c programm with gcc” в помощь).
  2. А что такое командная строка и как это запускать?
    После того как вы скомпилировали программу, у вас появился исполняемый файл. Вы можете запустить этот файл, написав его название в командную строку (в начале написав “./”, если вы работаете на MacOS/Linux). Также через пробел после названия файла можно передать аргументы командной строки (как передать аргументы командной строки в CLion). Также можете посмотреть поисковый запрос “c command line arguments”.
  3. Можно ли писать не на Си, а на С++?
    Нет, курс по C++ у вас будет в следующем семестре, в этом семестре можно писать только на Си.
  4. Какие библиотеки можно использовать?
    Можно использовать стандартную библиотеку и некоторые другие библиотеки, являющиеся де-факто типовым общепринятым решением для какой-либо задачи (например, unistd.h для парсинга аргументов командной строки).
  5. Где можно найти материал по Си и попрактиковаться в языке?
    Описание языка и всех функций стандартной библиотеки можно посмотреть здесь.
    Попрактиковаться можно на сайтах вроде exercism или HackerRank.

FAQ по лабораторной №1

  1. Что подаётся на вход программе, что является результатом работы программы?
    На вход подаётся ровно два аргумента командной строки, первый из которых – что мы хотим посчитать, второй – название файла, для которого мы хотим это сделать. Результатом выполнения программы должен быть вывод одного числа в стандартный поток вывода. Если на вход программе подано не два аргумента, либо если первый аргумент не соответствует ни одному из тех флагов, которые описаны в лабораторной, либо если переданного файла не существует, в общем, любая нестандартная ситуация, то программа должно вывести сообщение об ошибке и завершиться с ненулевым кодом.
  2. Можно ли передать больше одного типа подсчитываемого значения в программу?
    Нет, за один запуск можно передать только одно значение.
  3. Что считать словом, байтом, строкой?
    Словом нужно считать всё, что ограничено по обе стороны либо началом/концом строки, либо пробельным символом (поисковый запрос: whitespace character). Строкой считается нечто "зажатое" между символом новой строки/началом строки/концом строки. Пустые строки тоже валидные строки, их тоже надо считать. Для простоты можно считать, что все файлы закодированы в ASCII, поэтому один байт == один символ.
  4. Функции, которые могут пригодиться при написании лабораторной работы: fopen, fscanf, printf, fseek, ftell, strcmp, fclose, fgetc, fgets, getopt.
  5. Какой максимальный размер файла/строки в файле? Если у вас возник такой вопрос, вероятно вы пытаетесь прочитать целиком файл или строку из него и сохранить в массив. Подумайте, как обойтись без этого.

FAQ по лабораторной №3

  1. Не смотря на то, что в условии указан конкретный файл, с которым надо работать, следует воспринимать его как пример, и не завязываться на то, что вам подадут именно его. Вам могут подать логи, например, за август 95-го года, могут подать логи за последние 10 лет, и так далее. Единственное на что можно завязываться -- формат строки в файле. Решения, которые работают только с входным файлом из условия, получат максимум 5 баллов.
